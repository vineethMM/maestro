  16. case class UploadConfig(localIngestPath: String, hdfsLandingPath: String, localArchivePath: String, hdfsArchivePath: String, tablename: String, filePattern: String = "{table}?{yyyyMMdd}*", controlPattern: Regex = ControlPattern.default) extends Product with Serializable

    Configuration options for upload.

    Configuration options for upload.

    Files are copied from the local ingestion path to the three destination paths (HDFS landing path, local archive path, and HDFS archive path). When files are copied to a destination path the effective date of the upload are added to the path. For instance: the actual location of files on HDFS might be $hdfsLandingPath/<year>/<month>/<day>/<originalFileName>.

    localIngestPath

    The local ingestion path where source files are found.

    hdfsLandingPath

    The HDFS landing path where files are copied to.

    localArchivePath

    The local archive path where files are archived.

    hdfsArchivePath

    The HDFS archive path, where files are also archived.

    tablename

    The table name or file name in database or project.

    filePattern

    The file pattern, used to identify files to be uploaded. The default value is {table}?{yyyyMMdd}*.

    controlPattern

    The regex which identifies control files. The default value is au.com.cba.omnia.maestro.core.upload.ControlPattern.default.

  17. trait UploadExecution extends AnyRef

    Push source files to HDFS using upload and archive them.

    Push source files to HDFS using upload and archive them.

    See the example at au.com.cba.omnia.maestro.example.CustomerExecution.

    In order to run map-reduce jobs, we first need to get our data onto HDFS. upload copies data files from the local machine onto HDFS and archives the files. The user can also use findSources and uploadSources, which allows them to inspect the data files before uploading them.

    The user can run parseHeader, parseTrailer, and parseHT on data files to retrieve and validate their metadata.
